블로그를 다시 만들면서 가장 먼저 정한 원칙은 간단했다. 글은 데이터베이스가 아니라 저장소 안의 파일로 관리하고,
화면은 Next.js App Router 위에서 자연스럽게 조립한다.
별도의 CMS를 붙이기보다 글 하나가 하나의 디렉터리가 되고,
그 안에 본문과 이미지를 함께 두는 방식이다.

그래서 이 블로그의 글은 `src/content/posts/<slug>/post.mdx`에 놓인다.
썸네일은 같은 디렉터리의 `thumbnail.png`로 두고, 제목은 본문의 `#` heading에서 읽는다.
메타데이터는 MDX가 export하는 `metadata` 객체를 통해 description, tags, date만 명시한다.
제목을 메타데이터와 본문에 두 번 쓰지 않아도 되는 점이 마음에 든다.

## 글 하나의 단위

현재 글 디렉터리는 다음처럼 동작한다.

```text
src/content/posts/next-mdx-blog/
  post.mdx
  thumbnail.png
```

`post.mdx`는 글의 공개 API에 가깝다.
파일 상단에서 필요한 이미지를 import하고, `metadata`를 export하고,
그 아래에 일반 Markdown과 JSX를 섞어 본문을 작성한다.

```mdx
import Image from 'next/image';
import thumbnail from './thumbnail.png';

export const metadata = {
  description: '...',
  tags: ['MDX', 'Next.js'],
  date: new Date('2026-06-24'),
};

# 글 제목

<Image src={thumbnail} alt="..." />
```

이 구조에서는 글을 추가할 때 라우팅 코드를 만지지 않아도 된다.
새 slug 디렉터리를 만들고 `post.mdx`와 `thumbnail.png`를 넣으면
콘텐츠 로더가 파일 시스템에서 목록을 다시 만든다.

## 콘텐츠 로더

글 목록과 상세 글은 `src/lib/content/posts.ts`에서 읽는다. 역할은 작게 나뉘어 있다.

- `importList`는 `src/content/posts` 아래의 디렉터리 이름을 slug로 읽는다.
- `importImage`는 각 글의 `thumbnail.png`를 정적 이미지로 import한다.
- `importDocument`는 MDX 모듈을 import하고 `metadata`를 Zod 스키마로 검증한다.
- `importHeadings`는 MDX 원문을 파싱해서 첫 번째 `h1`을 제목으로, `h2` 목록을 본문 목차로 만든다.

여기서 중요한 점은 MDX를 단순한 문자열로 다루지 않는다는 것이다.
화면에 렌더링할 때는 Next.js의 MDX 모듈 시스템을 사용하고,
제목과 섹션을 뽑을 때는 `remark-parse`, `remark-mdx`, `unist-util-visit`를 사용한다.
글의 원천은 하나지만 필요한 형태로 읽는 방식은 분리되어 있다.

## 라우팅과 화면

상세 페이지는 `/posts/[slug]` 하나로 충분하다.
페이지 컴포넌트는 slug를 받아 `posts.get(slug)`를 호출하고,
없는 글이면 `notFound()`로 빠진다. 여기서 페이지가 직접 Markdown을 파싱하지 않는 점이 좋다.
라우트는 데이터를 가져오고, 헤더와 본문과 사이드바를 배치하는 일만 한다.

본문의 첫 번째 `#` 제목은 실제 문서 안에도 존재하지만,
MDX 컴포넌트 매핑에서 `h1`을 렌더링하지 않는다.
대신 `PostHeader`가 추출된 제목, 태그, 날짜를 보여준다.
이렇게 하면 글 파일은 Markdown 문서로서 자연스러운 형태를 유지하면서도
화면에서는 블로그 디자인에 맞는 헤더를 사용할 수 있다.

`h2`는 오른쪽 `On this Page` 목차에도 사용된다.
글을 작성할 때 큰 단락을 `##`로 나누면 페이지 구조가 자동으로 생긴다.
그래서 이 블로그의 글은 `##`를 기준으로 읽는 흐름을 먼저 잡고,
필요할 때만 `###`를 더하는 편이 잘 맞는다.

## MDX 컴포넌트

공유 MDX 컴포넌트는 `src/mdx-components.tsx`에서 연결한다.
지금은 단락, 제목, 목록, 인용문, 구분선 정도를 블로그 톤에 맞게 바꾸고 있다.
`rehype-slug`도 켜져 있어서 heading에는 id가 붙고,
사이드바 링크가 해당 섹션으로 이동할 수 있다.

MDX를 쓰는 이유는 Markdown만으로 충분한 글은 그대로 쓰고,
필요한 곳에서는 React 컴포넌트를 끼워 넣을 수 있기 때문이다.
이 글의 썸네일도 일반 Markdown 이미지가 아니라 `next/image`를 직접 사용한다.
이미지 최적화와 타입이 필요한 부분은 React가 맡고,
글의 대부분은 Markdown의 낮은 마찰을 유지한다.

> 글은 파일로 남기고, 표현은 컴포넌트로 조정한다. 이 둘을 억지로 합치지 않는 것이 MDX의 장점이다.

## 배포 산출물

콘텐츠 로더가 같은 데이터를 반환하므로 상세 페이지 외의 산출물도 자연스럽게 붙는다.
RSS는 `posts.list()`를 순회해 title, description, date를 넣고,
사이트맵도 같은 목록에서 slug와 date를 읽는다.
canonical URL과 Open Graph metadata는 상세 페이지의 `generateMetadata`에서 만든다.

결국 글 하나를 추가하는 행위가 여러 곳으로 전파된다.

1. `/posts/<slug>` 상세 페이지가 생긴다.
2. RSS 항목에 글이 추가된다.
3. 사이트맵에 글 URL과 수정 기준 날짜가 들어간다.
4. Open Graph와 canonical metadata가 글 단위로 생성된다.

글 작성자가 기억해야 할 것은 적다. `post.mdx`에는 본문과 `metadata`를 쓰고, 같은 디렉터리에 썸네일을 둔다.
나머지는 로더와 라우트가 처리한다.

## 아직 남은 일

지금 구조는 첫 글을 쓰기에 충분하지만, 블로그로 오래 쓰려면 몇 가지를 더 다듬어야 한다.

- 글 목록 페이지를 다시 열고 카드 컴포넌트를 채운다.
- 코드 블록과 표 스타일을 MDX 컴포넌트로 연결한다.
- 작성 중인 글과 공개 글을 구분할 방법을 정한다.
- 썸네일이 없는 글을 허용할지, 모든 글에 강제할지 결정한다.

특히 코드 블록은 개발 블로그에서 자주 등장하므로 곧 별도 컴포넌트로 다루는 편이 좋겠다.
지금은 콘텐츠 모델과 라우팅의 뼈대를 먼저 세웠고,
앞으로는 글을 쓰면서 필요한 표현을 하나씩 추가하는 방향이 맞아 보인다.

## 마무리

이 블로그는 큰 시스템이 아니다. 하지만 글을 추가하는 동선은 꽤 명확하다.
파일 시스템을 콘텐츠 저장소로 삼고, MDX를 문서와 컴포넌트의 접점으로 쓰고,
Next.js App Router가 페이지와 메타데이터를 만든다.

첫 번째 글로는 이 정도가 적당하다.
앞으로 이 블로그가 어떤 글을 담게 되든,
최소한 글을 쓰기 위해 블로그 자체와 싸우는 일은 줄어들 것이다.
